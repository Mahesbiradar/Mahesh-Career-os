## Section 1 — Career OS Overview

### What This System Is

Career OS is a structured 180-day execution system that converts daily study hours into verified software engineering mastery — tracked, evaluated, and managed by AI — so Mahesh can focus entirely on learning and building.

**One sentence:** Study and build every day; AI handles all tracking, scoring, planning, and scheduling.

## Section 4 — Study Session Protocol

### The 7-Step Session Protocol

Follow these steps for every topic, every day.

```
┌──────────────────────────────────────────────────────────┐
│  STEP 1: CALIBRATE                            [ 5 min ]  │
├──────────────────────────────────────────────────────────┤
│  Before opening any resource:                            │
│  Ask yourself: "What do I already know about this?"      │
│  Say it aloud or write it on paper.                      │
│  Note exactly what you cannot explain — those are        │
│  the gaps this session will fill.                        │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  STEP 2: STUDY                               [ 20–40 min]│
├──────────────────────────────────────────────────────────┤
│  Open ONE resource only (official docs first).           │
│  Read with a goal: "I need to understand X well          │
│  enough to build Y."                                     │
│  Maximum 40 minutes. If confused: implement anyway.      │
│  Confusion resolves through doing, not more reading.     │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  STEP 3: CLOSE ALL REFERENCES                 [ 30 sec ] │
├──────────────────────────────────────────────────────────┤
│  Close the documentation.                                │
│  Close YouTube. Close AI.                                │
│  Open only: code editor or blank notes file.             │
│  This is non-negotiable.                                 │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  STEP 4: IMPLEMENT                          [ 75–90 min ]│
├──────────────────────────────────────────────────────────┤
│  Build the mini-task from today's AI plan.               │
│  Write every line yourself.                              │
│  ⚠ AI IS OFF. No exceptions.                            │
│  When stuck: think 5 min → check specific doc syntax     │
│  → try again.                                            │
│  After 15 consecutive minutes stuck on ONE thing:        │
│  AI is permitted (Mode 2 — Explainer). ADS increases.   │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  STEP 5: DEBUG AND FIX                     [ until done ]│
├──────────────────────────────────────────────────────────┤
│  When something breaks:                                  │
│  → Read the full error message before doing anything.    │
│  → Form a hypothesis about the cause.                    │
│  → Test the hypothesis.                                  │
│  → Attempt fix.                                          │
│  15-minute rule: independent first. AI only after.       │
│  The debugging process IS the learning.                  │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  STEP 6: SELF-CHECK                           [ 5 min ]  │
├──────────────────────────────────────────────────────────┤
│  Ask yourself (out loud):                                │
│  "Without AI or notes right now — can I explain          │
│  this topic clearly AND implement its core?"             │
│                                                          │
│  YES (both) → Log ADS as-is. Topic on track.            │
│  EXPLAIN only → Downgrade Level Guess by 1 ML.           │
│                  Tomorrow: more implementation.           │
│  NEITHER     → Set Level Guess to ML-2 or lower.         │
│                  Log blocker. AI adjusts tomorrow.       │
└──────────────────────────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────┐
│  STEP 7: EVIDENCE AND LOG                     [ 5 min ]  │
├──────────────────────────────────────────────────────────┤
│  Code topics: git commit -m "topic: what was built"      │
│  Theory topics: save note (150+ words, own words)        │
│  Then: fill End-of-Day Report (Section 5).               │
└──────────────────────────────────────────────────────────┘
```

### Theory Topic Variant (OS, CN, DBMS, Communication)

For topics where no code is produced, replace Steps 4–5 with:

```
STEP 4T: WRITE-TO-UNDERSTAND
  After closing all references:
  Write a 150–250 word explanation of the topic in your own words.
  Write as if explaining to another engineer who has never seen it.
  Do NOT look at notes while writing.
  If you cannot write 150 words from memory:
    → Re-read source (Step 2 again, 15 min max)
    → Close it. Try again.
  The writing IS the implementation for theory topics.
```

### AI Usage Rules During Sessions

| Moment | AI Mode | Permitted |
|--------|---------|-----------|
| Block 1 — Study | Mode 2 (Explainer) | After 40-min reading cap, if still stuck |
| Block 2 — Implement | Mode 0 (OFF) | Nothing. Absolute. |
| Debugging | Mode 0 first | After 15 min of independent struggle → Mode 2 |
| Self-Check | Mode 1 (Evaluator) | Ask AI to test you — never to answer for you |
| Evidence creation | Mode 0 (OFF) | All writing and coding is Mahesh's own |

**The Explain-Back Rule (mandatory for every AI explanation):**
After any AI explanation → close AI → write the explanation in your own words immediately → verify it captures the concept → only then proceed.

### ADS Reference (fill this accurately — it drives all scoring)

| ADS | Meaning |
|-----|---------|
| 0 | Fully independent — zero AI use during the session |
| 1 | Minimal AI — used for 1–2 quick lookups (syntax, not concepts) |
| 2 | Moderate AI — used for explanation after genuine struggle |
| 3 | High AI — needed AI to understand core concepts or debug |
| 4 | Fully dependent — AI wrote or heavily guided implementation |

**Rule:** ADS must be honest. An inflated ADS (reporting lower than actual) corrupts all downstream scoring, evaluation, and planning. There is no penalty for a high ADS — only for dishonesty.

## Section 9 — Emergency Recovery Protocol

### Recovery by Disruption Type

---

#### Missed 1 Day

```
WHAT HAPPENED: 0 hours studied. No log filed.

RECOVERY (takes 2 minutes):
1. File a zero-log immediately:
   TOPIC: None
   BUILT: Day missed — [one-word reason: illness/schedule/energy]
   ADS: N/A
   BLOCKER: Day missed
   LEVEL GUESS: N/A

2. Send it via Daily Evaluation Prompt.

3. Resume normal schedule tomorrow.
   Do NOT add extra hours to compensate.
   Do NOT feel guilty — log it and move on.

RULE: One missed day changes nothing. Three missed days in one
      week triggers the Missed 3 Days protocol.
```

---

#### Missed 3 Days

```
WHAT HAPPENED: 3+ consecutive or near-consecutive days missed.

RECOVERY (takes 15 minutes):
1. File zero-logs for each missed day (2 min).

2. Send Full System Status Prompt:
   "Run a system status. I missed [N] days due to [reason].
   Last active session log: [paste]. What is my current CRS,
   domain health, and what should my first session back cover?"

3. Resume with a REDUCED scope day:
   - Block 1: Re-read last topic studied (30 min, no new content)
   - Block 2: Rebuild last mini-task from memory (no reference)
   - DSA: 30 min only
   - Admin: Normal
   No new topics on the first day back.

4. Return to full schedule on Day 2 after the break.

PHASE IMPACT: If missed days caused a week to fall below 4 study
              days, extend that sprint's goals into the next week.
              Do not rush to "catch up" — this causes F-7 (burnout).
```

---

#### Missed 7 Days

```
WHAT HAPPENED: A full week or more missed due to illness,
               emergency, or major life event.

RECOVERY (takes 3 days to ramp back):
1. File zero-logs for all missed days (5 min total).

2. Send Monthly Review Prompt for a situation check.

3. Three-day ramp-back schedule:
   Day 1 back: 3 hours — revision only (no new topics)
               Re-check last 5 completed topics via PROMPT 8
   Day 2 back: 4 hours — light new content + revision
   Day 3 back: 5 hours — normal execution resumes
   Day 4+: Full 6-hour schedule

4. AI recalibrates phase timeline based on days lost.
   The 180-day horizon shifts accordingly.
   A 7-day gap adds approximately 8–10 days to the timeline
   (because of ramp time + re-establishing momentum).

WHAT NOT TO DO:
- Do not study 10 hours on Day 1 back.
- Do not skip the ramp period.
- Do not panic about the CRS — it will recover.
```

---

#### Burnout

```
SYMPTOMS: Consistently < 3 hours/day for 3+ consecutive days
          despite no external emergency. Resistance to opening
          the laptop. Feeling that the effort is pointless.

RECOVERY:
Step 1: Take 2 complete rest days. No study. No logs.
        No reading about software engineering.
        This is not optional — burnout compounds without rest.

Step 2: On Day 3, send Full System Status Prompt.
        See the actual CRS number. Progress is usually more
        real than it feels during burnout.

Step 3: Return with 3-hour sessions for 3 days.
        One domain only — whichever feels most engaging.
        No pressure on volume.

Step 4: Identify the root cause:
        - Topic too hard → Level 1 Escalation
        - No visible progress → CRS Bottleneck Analysis
        - Isolation → Add daily Communication session with
          a peer or mentor (even one person)
        - Unclear goal → Re-read Section 1 of this guide

Step 5: Return to full schedule only when 3 consecutive
        4-hour sessions have been completed without resistance.

PREVENTION: Burnout is caused by intensity without recovery.
            One lighter day per week (3 hours, one domain only)
            is better than grinding 7 days then collapsing.
```

---

#### Freelance Work Interruption

```
SITUATION: Paid freelance work demands time from the study schedule
           for a defined period (days to weeks).

RECOVERY PLAN:
Minimum viable Career OS during freelance work:

DAILY MINIMUM (1.5 hours):
  Communication: 15 min speaking practice (non-negotiable)
  Python: 30 min reading only (no implementation)
  DSA: 30 min (reduced from standard)
  Log: 5 min (file a partial log, honest about time)

FULL CAREER OS: Resumes when freelance work ends.

RULE: Do not attempt full Career OS during heavy freelance
      periods — partial effort across 5 hours produces worse
      results than focused effort across 1.5 hours.
      Better to maintain the habit at minimum than to burn out
      trying to do both at full intensity.

When freelance ends: Use the Missed 3 Days ramp protocol.
```

## Section 11 — Operating Rules

> These are the laws of the system. Non-negotiable. No exceptions.

---

### Rule 1: Never Skip Evidence

**The rule:** Every session that studies a topic must produce an artifact before the log is filed. Always.

**What counts:**
- Code: GitHub commit with a topic-specific message
- Theory: 150+ word written explanation in your own words, saved anywhere

**What does not count:**
- Screenshots of AI output
- Copy-pasted code from tutorials
- Notes that say "I understand this"
- A topic marked Complete with no artifact

**Why:** Evidence is the only honest record of what was understood. Without it, the system has no data. The log becomes fiction. The plans become wrong.

---

### Rule 2: Never Mark Complete Without Implementation

**The rule:** A topic cannot be marked COMPLETE unless a Full Evaluation has been run (Prompt 3) and an artifact exists.

**No exceptions for:**
- Topics that "feel" understood
- Topics that were covered in a tutorial
- Topics that were explained well by AI
- Topics from previous life experience (telecom background does not exempt any syllabus topic)

**Why:** The evaluation is the only way to distinguish familiarity from proof. The artifact is the only way to verify that implementation happened independently.

---

### Rule 3: Never Use AI Before Attempting the Solution

**The rule:** The 15-minute rule applies to every debugging situation. Attempt independently for 15 minutes before AI assistance.

**What "attempt" means:**
- Read the full error message
- Form a hypothesis
- Try something
- Try something else
- After 15 minutes: AI is permitted

**What "attempt" does not mean:**
- Reading the error, feeling lost, and immediately asking AI
- Googling the error and immediately asking AI to explain the Google result

**Why:** Every interview will require debugging without AI. The 15 minutes per day builds the muscle. Skipping it means arriving at interviews unable to debug independently.

---

### Rule 4: DSA Remains Separate

**The rule:** DSA is never logged in Career OS. DSA topics never appear in Career OS reports. DSA progress never appears in CRS. Career OS AI prompts never ask about DSA performance.

**DSA block:** 60–75 minutes, first slot of the day, governed entirely by the DSA system.

**Why:** Two systems tracking the same domain creates confusion and duplication. Each system is simpler and more effective when it has clean responsibility boundaries.

---

### Rule 5: Communication Practice Is Daily

**The rule:** 15 minutes of spoken communication practice happens every day. Not 5 days per week. Not "when there's time." Every day.

**Minimum viable on low-energy days:**
- Deliver self-introduction twice (2 min)
- Pronunciation drill of 12 terms (2 min)
- One question: "Explain [today's topic] as if to a non-technical interviewer" (3 min)

**Why:** Communication compounds through daily repetition. 15 minutes done 7 days a week is more effective than 90 minutes done once a week. Missing this for a week is the fastest way to fail an interview on the soft-skills dimension after passing the technical rounds.

---

### Rule 6: Consistency Beats Intensity

**The rule:** A 5-hour study day done consistently for 6 months beats a 10-hour study day done sporadically. Never sacrifice tomorrow's session for today's extra hour.

**In practice:**
- Stop studying when tired, not when the plan is complete
- Rest days are planned, not failures
- A 3-hour partial day is logged honestly and continued — not made up with a 9-hour day tomorrow

**Why:** This is a 180-day system. It is designed for the long run. Burning out at Day 60 produces nothing. Steady effort at Day 180 produces employment.

---

### Rule 7: Honest Logs Only

**The rule:** Every field in every log must reflect reality.

**Specifically:**
- ADS reflects actual AI usage — not what would look good
- Level Guess reflects actual capability — not what was hoped
- BLOCKER reflects actual obstacles — not "none" when there was one
- TIME STUDIED reflects actual hours — not the plan

**Why:** The AI uses the logs to generate every plan, every evaluation, every intervention. An inflated log generates a plan that is too advanced. A deflated log generates a plan that is too easy. Both corrupt the system. Honesty is not a virtue here — it is a technical requirement.

---

### Rule 8: The Plan Is Read Before Studying Begins

**The rule:** The AI-generated morning plan is read before opening any study material. Always. This takes 2 minutes.

**Why:** Without reading the plan, the day's study direction is random. The AI plan contains: the correct next topic in the syllabus sequence, the specific mini-task, the revision items due, and any flags from yesterday's session. Skipping it wastes the planning work the AI already did.

---

### Rule 9: Never Switch Domains Mid-Session When Stuck

**The rule:** When Block 2 implementation is failing, the response is to reduce scope (do a smaller part of the task), not to switch to a different domain.

**Why:** The failure in implementation is almost always the learning moment. Switching domains avoids it. The avoided learning becomes a permanent gap. Every time a stuck moment is escaped by switching, the stuck topic gets harder to return to.

---

### Rule 10: The Log Is Filed Even on Bad Days

**The rule:** If any studying happened — even 1 hour — a log is filed. If no studying happened, a zero-log is filed.

**The zero-log:**
```
TOPIC: None | BUILT: Day missed — [reason]
ADS: N/A | BLOCKER: Day missed | LEVEL GUESS: N/A
```

**Why:** Continuous logs give the AI the data it needs to adjust for disruptions. A missing log creates a gap in the record that causes tomorrow's plan to be generated from stale data.
